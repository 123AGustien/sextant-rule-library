# DC-004 — Compute Load Spike

## Rule Information

Rule ID: DC-004

Domain: Data Centre Resilience

Category: Compute / Workload Stress

Status: Active

Version: 1.0

Repository: Sextant Rule Library

---

## Purpose

Evaluate data centre resilience under sudden increases in compute demand and assess cascading impacts across system performance, infrastructure stability, and dependent domains.

---

## Scenario

A rapid and unexpected surge in computational demand occurs due to workload spikes, external traffic surges, batch processing events, or cascading upstream system triggers. This results in increased CPU, memory, and storage pressure across infrastructure.

The scenario evaluates:

- CPU utilisation
- Memory consumption
- Storage I/O demand
- Cluster load balancing
- Virtual machine/container density
- Application throughput

---

## Primary Indicators

- CPU utilisation (%)
- Memory utilisation (%)
- Disk I/O rates
- System queue depth
- Container/VM scaling events
- Load balancer distribution
- Application response time

---

## Risk Levels

### GREEN

Normal compute load conditions.

System resources operate within expected thresholds.

No contingency actions required.

---

### YELLOW

Early signs of elevated compute demand.

Monitor:

- CPU usage trends
- Memory consumption
- Scaling activity
- Application latency

Action:

Prepare for autoscaling and optimise workload distribution.

---

### ORANGE

Sustained compute load pressure.

Indicators:

- High CPU utilisation
- Memory pressure increasing
- Delayed scaling response
- Application slowdown

Potential Cascade:

Compute Load Spike

↓

Resource Saturation

↓

Performance Degradation

↓

Service Latency Increase

Action:

Activate scaling procedures, redistribute workloads, and prioritise critical applications.

---

### RED

Critical compute saturation.

Indicators:

- Resource exhaustion
- System throttling
- Service outages
- Failed scaling operations

Potential Cascade:

Compute Overload

↓

System Instability

↓

Application Failure

↓

Infrastructure Stress

↓

Cross-Domain Systemic Risk

Action:

Initiate emergency workload shedding, failover systems, and disaster recovery protocols.

---

## Cross-Domain Dependencies

Data Centre (DC)

↓

Infrastructure (INF)

↓

Cyber (CYB)

↓

Financial (FIN)

Compute stress may propagate into service degradation, cybersecurity monitoring delays, and financial system impacts.

---

## Contingency Actions

- Monitor compute utilisation continuously
- Enable autoscaling systems
- Prioritise critical workloads
- Shed non-essential processing
- Review system capacity planning
- Escalate to incident response when thresholds are exceeded
